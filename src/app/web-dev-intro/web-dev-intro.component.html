1. Overview of Web Development

Web development is the building of software designed to be used on Internet Browsers. This can mean web pages on the Internet, or on a company's private network, or even just hosted locally on your computer. To get a better sense of what web development is, let's go through some things that aren't web development...
  • Desktop applications, like .exe files on Windows or .dmg files on Mac. These do not run on Internet Browsers, and so they rely on the Operating System's specific tools to be built. This is also why it is difficult to have cross-platform software.
  • Scripts, like Python scripts or shell scripts. Scripts are designed to do very specific tasks, and often have very basic user interfaces. Web and front-end development is targeted mostly at the general public as end users, so the interface must be easy to use.
  • Google Sheets/Excel, these are both great tools for building applications, especially as building basic applications is relatively straightforward. However, building a tool inside such a tool also has severe limitations, e.g. difficulties with scaling to a larger size, version control, etc.
  • Blogging sites e.g. WordPress, Tumblr, although this is a grey area. Blogging sites usually allow you to define static content, but it can be difficult to define dynamic content (e.g. a form which users can submit, or complex UI elements which change based on user input). You usually have freedom to use your own HTML/CSS freely otherwise.
Even though these examples are not web development, each of them are legitimate tools and solutions for problems in their own right. Web development is a tool, and it's important to understand what the capabilities and limitations of any tool are. Some pros of web development are:
  • A very visual and user-friendly interface for end users
  • Compatibility across many different browsers and devices (e.g. Chrome, Firefox, Microsoft Edge, Safari, on computers, tablets, phones, etc.)
  • Applications can be run on the user's device, so it is possible to shift some computational burden away from the server hosting the website
  • Front-end development is very common, so it is relatively easy to find developers to build web applications (relative to perhaps, building a desktop Linux application)
And some cons...
  • There is a substantial learning curve! Building a web application is not as easy as assembling a basic application in Excel or Google Sheets
  • Building web applications can be time-consuming! Before building a web application, ask yourself, is it worth it? If you are building a site for your portfolio which only a few people will see, and you're only looking to get the job done quickly, perhaps it's better to use a blogging website or a low-code platform.
  • Hosting websites usually costs money. There are ways to host websites for free, such as with GitHub Pages, but most web applications will need to be hosted on a provider such as AWS, Azure or Google Cloud, or on your own server. And as an extreme case, malicious actors can overwhelm your site and drive up your upkeep costs!

2. Basic Concepts

2.1. HTML, CSS, JS

  • HTML (Hypertext Markup Language) defines what is shown on a page, and how it is displayed.
  • CSS (Cascading Style Sheets) is used to format how HTML is shown (e.g. changing colour, font size, etc.).
  • JS (JavaScript) is used to make HTML and CSS interactive and dynamic, e.g.
    • Read and process text inputs from users
    • Send and fetch data to/from servers
    • Create more advanced formatting and layout which may be difficult with HTML/CSS alone

2.2. Front-end and Back-end

  • The front-end (or client-side) is the part of the application which the end user interacts with.
  • The back-end (or server-side) is everything else.
Life Cycle of a Web Page
  1. Alice opens enters a URL into her browser.
  2. The browser sends a HTTP request to the address given by the URL, which is received by a server (or not).
  3. The server sends back all the necessary files (HTML, CSS, JS, images, other assets etc.) needed for the requested page via another HTTP request.
  4. The browser receives the files, and renders the raw HTML/CSS/JS into the web page for Alice to use.
  5. Alice interacts with elements on the page, which trigger JavaScript functions, that can alter both the internal state of the page, and what is displayed on the page.
  6. Upon Alice clicking a button to submit some data, some JavaScript runs to send a HTTP request with the submitted data to the server.
  7. The server sends back a HTTP request to confirm it successfully received the sent data.
What about very complex web pages?
When developing applications, breaking code into smaller logical sections helps keep the code maintainable. This is usually done by separating HTML/CSS/JS files into multiple files. A bundler (such as webpack or Vite) can be used to re-compile these multiple files into individual HTML/CSS/JS classes for easy deployment and use by end users.
What is TypeScript?
TypeScript is a superset of JavaScript which adds static typing to JavaScript, and is usually written in .ts files. TypeScript can be compiled to JavaScript and bundled the same as JavaScript can be.

3. Common Frameworks and Libraries

JavaScript Frameworks

Front-end frameworks are collections of code which can be used to create front-end applications in a quick or efficient way. Frameworks can differ in the amount of structure/flexibility are allowed, the syntax used, and other specific functionality, amongst others. Some common frameworks are listed below (note however that some people don't call all of these 'frameworks' - we will call them frameworks for ease of explanation):
  • Angular: One of the most widely used front-end frameworks. Supported by Google, and provides a lot of structure to organise code and functionality (good at enforcing good practices, especially in very large applications, but can feel limiting or overly verbose at times).
  • React: One of the most widely used front-end frameworks. Supported by Facebook, and is highly modular and flexible (less enforced structure can mean difficulty in keeping code organised).
  • Vue: A well-known framework inspired by Angular, but with more flexibility. However, not as popular as React or Angular, and less major corporate backing.
  • Svelte: A well-known framework for writing front-end components. Lightweight in rendering and file size, but less popular than the above options, and less major corporate backing.

CSS Frameworks

CSS Frameworks are utilities which simplify how styling and layout is handled. Some common such frameworks are:
  • Bootstrap: Arguably the most well-known CSS framework. Provides a small, but versatile and configurable list of CSS classes which can be used in a modular way throughout the application. Beginner-friendly, as even minimal usage can produce good results.
  • Tailwind: A newer, but still well-known CSS framework. Provides a large, comprehensive and granular list of CSS classes. Much more verbose and requires more knowledge of basic CSS, but scales very well to complex applications due to being very modular.

Component Libraries

Component Libraries are - as their name suggest - libraries of components, which can be used with relatively minimal setup. Your choice of specific component library may depend on your chosen JavaScript framework, although the largest component libraries have equivalents in each major framework. Some examples are:
  • NgBootstrap: Bootstrap also has a component library! If you are using Bootstrap for its CSS utilities, it is relatively easy to also use Bootstrap's components (although you can still choose other component libraries.
  • Angular Material: This is the Angular library for the Material Design specification, a design system from Google. A lot more modern than Bootstrap, but for simplicity we will stick with NgBootstrap here.
  • Tailwind UI: Tailwind's UI component library. Similarly to NgBootstrap, works well with its associated CSS framework. However, this is not a free component library, whereas Tailwind CSS is free to use!
